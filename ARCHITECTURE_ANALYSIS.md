## 1. СТРАТЕГИЧЕСКАЯ ОЦЕНКА: Правильный ли это подход?

### 1.1. Короткий ответ

**Да, это правильный подход.** Трёхслойная архитектура (RAW → Normalized → Clean → 1С) — это по сути **accounting middleware**, которое решает фундаментальную проблему: традиционные системы учёта (1С, SAP) проектировались для ручного ввода, а не для автоматической обработки разнородных данных с участием AI.

Но есть **4 принципиальные коррекции**, которые превратят хорошую архитектуру в отличную.

### 1.2. Почему этот подход верный

**Проблема, которую вы решаете:**

```
БЫЛО (классика):                    СТАЛО (ваш подход):

Данные → Бухгалтер → 1С             Данные → Система → AI + Человек → 1С
         (ручной ввод,                        (автопарсинг,   (сверка,      (только
          ошибки,                               нормализация)   аналитика)    печать)
          задержки)
```

Вы по сути строите **"мозг" перед "принтером"**. Это абсолютно правильно, потому что:

1. **AI-агенты работают с данными, не с интерфейсами.** Неважно, насколько удобен UI 1С — для AI нужна машиночитаемая структура данных с API. Ваш слой 2 (Normalized) — это именно это.

2. **Иммутабельность = доверие.** В учёте нельзя «потерять» данные. Ваш append-only RAW слой + audit trail корректировок — это правильный паттерн для любого типа учёта, не только ГСМ.

3. **Разделение "грязных" и "чистых" данных** — это не просто техническое решение, это бизнес-правило. Бухгалтер не должен заниматься выверкой — это задача системы и менеджера.

4. **Расширяемость через JSONB** — новые типы документов (зарплата, аренда, логистика) не требуют миграции схемы. Это критично для бизнеса, который растёт и добавляет направления.

### 1.3. Четыре принципиальные коррекции

#### Коррекция 1: Semantic API — недостаточно. Нужен **Document Graph**

Промпт предлагает Semantic API для AI-агентов (find_shift_reports, get_shift_summary). Это хорошо для первой версии, но **AI 2025-2026 года уже умеет работать напрямую с SQL и JSONB**.

**Вместо** жёстких endpoints предлагаю:

```
Уровень 1: SQL-доступ через read-only реплику (AI пишет запросы сам)
Уровень 2: Semantic API (предвычисленные агрегаты для типовых задач)
Уровень 3: Document Graph (связи между документами)
```

**Document Graph** — это граф связей между всеми документами:

```
ТТН-1234 ──поступление──→ Резервуар-1 ──хранение──→ Смена-6890
                                                        │
                                    ┌───────────────────┤
                                    ↓                   ↓
                              Чек-001...Чек-150    Сменный отчёт
                                    │                   │
                                    ↓                   ↓
                              Банк.реестр        Отчёт о розн. продажах
                                                        │
                                                        ↓
                                                   XML → 1С
```

AI-агент, имея граф, может:
- Найти **причину** расхождения (а не просто факт)
- Предложить **корректировку** с обоснованием
- Построить **цепочку доказательств** для аудитора

**Реализация:** таблица `document_links` (source_id, target_id, link_type, metadata).

#### Коррекция 2: Не "ReconciliationEngine" — а **Rule Engine с самообучением**

Текущая архитектура: набор захардкоженных проверок (cash_vs_receipts, bank_vs_acquiring и т.д.).

Проблема: при добавлении нового типа учёта (например, зарплата) нужно писать новые проверки с нуля.

**Предлагаю:** Декларативный Rule Engine:

```python
# Правила сверки — это данные, а не код
RECONCILIATION_RULES = [
    {
        "id": "cash_vs_receipts",
        "name": "Касса ККТ = сумма чеков (наличные)",
        "doc_type": "shift_report",
        "source_path": "metadata.payments.cash",
        "target": {
            "doc_type": "receipt",
            "aggregate": "SUM(metadata.amount) WHERE metadata.payment_type = 'cash'"
        },
        "thresholds": {
            "ok": 0,           # точное совпадение
            "warning": 100,     # до 100₽ — предупреждение
            "critical": 500     # от 500₽ — критичное
        },
        "auto_correct": {
            "enabled": False,   # пока ручная корректировка
            "ai_suggest": True  # но AI предлагает исправление
        }
    },
    # ... другие правила
]
```

**Самообучение:** AI анализирует историю корректировок и:
- Предлагает новые правила (обнаружил паттерн → "всегда расхождение по ДТ в ночную смену на АЗС-208")
- Корректирует пороги (warning/critical) на основе статистики
- Автоматически применяет корректировки с высокой уверенностью (>95%)

#### Коррекция 3: Слой 3 (Clean) — не просто чистовые документы, а **Accounting Kernel**

Текущее описание слоя 3 — это генератор XML для 1С. Это слишком узко.

Слой 3 должен быть **ядром учёта**, которое:
- Ведёт **план счетов** (не зависит от 1С!)
- Формирует **проводки** по правилам (настраиваемый маппинг)
- Генерирует **любой** выходной формат (1С XML, SAP IDoc, Excel, API)
- Поддерживает **несколько учётных политик** (бухгалтерский + управленческий + налоговый)

```
Слой 2 (Verified) → Accounting Kernel → 1С XML
                                       → Управленческий отчёт
                                       → Налоговый регистр
                                       → Excel сводная
                                       → AI Summary
```

Это критично для масштабирования на другие типы учёта. Зарплата, аренда, логистика — все проходят через один Accounting Kernel с разными маппингами.

#### Коррекция 4: Event-Driven вместо Request-Response

Текущая архитектура подразумевает линейный pipeline: загрузка → парсинг → сверка → верификация → экспорт.

Для production-системы с 100+ АЗС нужна **событийная архитектура:**

```
[STS API] ──webhook──→ [Event Bus] ──→ [Parser Worker]
[Bank CSV] ──upload──→ [Event Bus] ──→ [Reconciliation Worker]
[1С Feedback] ──xml──→ [Event Bus] ──→ [Feedback Worker]

Event Bus = PostgreSQL LISTEN/NOTIFY (простой вариант)
          = Redis Streams (средний)
          = RabbitMQ/Kafka (масштабный)
```

**Почему важно:**
- Параллельная обработка (100 смен одновременно)
- Устойчивость к сбоям (retry, dead letter queue)
- AI-агент может подписаться на события ("уведоми меня о критических расхождениях")
- Каждый шаг — независимый worker, можно масштабировать отдельно

### 1.4. Сравнение с альтернативами

| Подход | Плюсы | Минусы | Вердикт |
|--------|-------|--------|---------|
| **Ваш (3 слоя + JSONB)** | Гибкость, AI-ready, расширяемость | Требует дисциплины с JSONB | **Правильный выбор** |
| Всё в 1С | Бухгалтеры знают | Нет API для AI, сложно масштабировать | Не подходит |
| Чистый NoSQL (MongoDB) | Максимальная гибкость | Нет транзакций, сложные JOIN-ы | Избыточно |
| Чистый SQL (без JSONB) | Строгая схема | Миграции при каждом новом типе | Слишком жёстко |
| ERP (SAP/Oracle) | Промышленный | Дорого, сложно, нет AI-first | Не ваш масштаб |

## 2. АРХИТЕКТУРНЫЕ РЕШЕНИЯ

### 2.1. Гибридная модель: когда колонка, когда JSONB

**Правило:** Если поле участвует в WHERE, JOIN, GROUP BY, ORDER BY — это колонка. Всё остальное — JSONB.

```
┌─────────────────────────────────────────────────────────┐
│                    КОЛОНКИ (индексируемые)               │
│  id, doc_type, doc_date, station_id, counterparty_id,   │
│  status, reconciliation_status, created_at, verified_at  │
├─────────────────────────────────────────────────────────┤
│                    JSONB metadata                        │
│  Всё специфичное для конкретного типа документа:        │
│  shift_number, operator, sales_by_fuel, payments,       │
│  tank_balances, reconciliation details, ...              │
├─────────────────────────────────────────────────────────┤
│              GENERATED COLUMNS (из JSONB)                │
│  Часто запрашиваемые поля из metadata:                  │
│  total_amount, operator_name, shift_number               │
│  (вычисляются автоматически, индексируются)              │
└─────────────────────────────────────────────────────────┘
```

**Критерий миграции из JSONB → колонку:**
- Поле используется в >30% запросов → вынести в колонку
- Поле участвует в сверке → вынести в колонку
- Поле нужно для отчётности → Generated Column

### 2.2. Индексация

```sql
-- B-tree: для точных совпадений и диапазонов
CREATE INDEX idx_docs_type_date ON documents (doc_type, doc_date);
CREATE INDEX idx_docs_station_date ON documents (station_id, doc_date);
CREATE INDEX idx_docs_status ON documents (status) WHERE status != 'verified';

-- GIN: для поиска ВНУТРИ JSONB
CREATE INDEX idx_docs_metadata ON documents USING GIN (metadata jsonb_path_ops);
-- jsonb_path_ops быстрее стандартного GIN для операторов @>, но не поддерживает ?/? |

-- Частичные GIN-индексы (экономят место):
CREATE INDEX idx_shift_reports_metadata
  ON documents USING GIN (metadata jsonb_path_ops)
  WHERE doc_type = 'shift_report';

-- Generated columns для частых запросов:
ALTER TABLE documents ADD COLUMN total_amount NUMERIC
  GENERATED ALWAYS AS ((metadata->>'total_amount')::numeric) STORED;
CREATE INDEX idx_docs_total_amount ON documents (total_amount);
```

**Рекомендация по производительности:**
- GIN-индексы тяжелы при записи (замедляют INSERT на 20-40%)
- Для таблицы raw_data GIN НЕ нужен (данные пишутся, не ищутся)
- Для documents — GIN оправдан (читаются чаще, чем пишутся)

### 2.3. Партиционирование

При 100+ АЗС × 2 смены/день × 365 дней = ~73 000 документов/год на слой 2.

```sql
-- Партиционирование по месяцам
CREATE TABLE documents (
    id UUID NOT NULL,
    doc_date TIMESTAMPTZ NOT NULL,
    doc_type TEXT NOT NULL,
    -- ...
) PARTITION BY RANGE (doc_date);

-- Автоматическое создание партиций (pg_partman или вручную):
CREATE TABLE documents_2026_01 PARTITION OF documents
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE documents_2026_02 PARTITION OF documents
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

-- Партиция raw_data отдельно (растёт быстрее):
CREATE TABLE raw_data (
    id UUID NOT NULL,
    received_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    -- ...
) PARTITION BY RANGE (received_at);
```

**73K docs/year** — это немного для PostgreSQL. Партиционирование оправдано, но не критично. Реальная польза — при архивации старых данных (DETACH PARTITION → archive).

### 2.4. Валидация JSONB

**Двухуровневая стратегия:**

```
Уровень 1: CHECK constraint (минимальная защита от мусора)
Уровень 2: Pydantic model (полная валидация в Python)
```

```sql
-- Уровень 1: базовый CHECK
ALTER TABLE documents ADD CONSTRAINT check_metadata_not_null
    CHECK (metadata IS NOT NULL AND metadata != '{}'::jsonb);

-- Для конкретных типов:
ALTER TABLE documents ADD CONSTRAINT check_shift_report_fields
    CHECK (
        doc_type != 'shift_report'
        OR (
            metadata ? 'shift_number'
            AND metadata ? 'payments'
            AND metadata ? 'sales_by_fuel'
        )
    );
```

```python
# Уровень 2: полная валидация в Python
from pydantic import BaseModel, validator

class ShiftReportMetadata(BaseModel):
    shift_number: int
    shift_start: datetime
    shift_end: datetime
    operator: str
    sales_by_fuel: dict[str, FuelSalesItem]
    payments: PaymentsSummary
    tank_balances: list[TankBalance]

    @validator('shift_end')
    def end_after_start(cls, v, values):
        if v <= values.get('shift_start', v):
            raise ValueError('shift_end must be after shift_start')
        return v
```

## 3. БИЗНЕС-ЛОГИКА И СВЕРКИ

### 3.1. Матрица сверок (расширенная)

Ваша матрица из FUEL_ACCOUNTING_SYSTEM.md хорошая. Добавляю классификацию:

| Сверка | Тип | Автоматизация | Порог warning | Порог critical |
|--------|-----|---------------|---------------|----------------|
| Операции ↔ Сменный отчёт (литры) | Точная | 100% авто | >0.1% | >1% |
| Касса ↔ Сменный отчёт (₽) | Точная | 100% авто | >0 ₽ | >100 ₽ |
| Банк ↔ ККТ безнал | Точная | 100% авто | >0 ₽ | >100 ₽ |
| Топл.карты ↔ Операции | Поштучная | 95% авто | 1 непарная | >3 непарных |
| Онлайн ↔ Операции | Поштучная | 95% авто | 1 непарная | >3 непарных |
| ТТН ↔ Факт приёмки | Нормативная | 100% авто | >0.3% (масса) | >0.5% |
| Остатки резервуаров | Расчётная | 100% авто | >0.3% | >0.5% |

**Ключевое:** Пороги — это **конфигурация, а не код**. Хранить в таблице `reconciliation_thresholds` или в справочнике Supabase. Менеджер может менять пороги без релиза.

### 3.2. Workflow корректировок

```
Расхождение обнаружено
        │
        ▼
┌─ AI-агент анализирует ─────────────────────────────┐
│  • Это типовое расхождение? (паттерн в истории)     │
│  • Какова вероятная причина?                         │
│  • Предлагаемая корректировка + уверенность          │
└─────────────────────────────────────────────────────┘
        │
        ├── Уверенность > 95% → Автокорректировка + лог
        │                        (менеджер видит в ленте)
        │
        ├── Уверенность 70-95% → Предложение менеджеру
        │                         (approve/reject одним кликом)
        │
        └── Уверенность < 70% → Ручной разбор
                                  (менеджер исследует)
```

**Типичные паттерны автокорректировки (для обучения AI):**
- Возврат по карте: сумма минус → парная операция с тем же чеком
- Тестовая транзакция: 1 литр / 1₽ → исключить из сверки
- Ночная смена пересечение: операция в 00:01 → отнести к предыдущей смене
- Потери в норме: отклонение < 0.3% → списание по нормативу

## 4. ИНТЕГРАЦИЯ С 1С

### 4.1. Выбор формата обмена

| Формат | Для чего подходит | Ограничения |
|--------|------------------|-------------|
| **CommerceML 2.x** | Каталоги товаров, простые заказы | НЕ поддерживает проводки, субконто |
| **EnterpriseData** | Полный обмен (1С:Предприятие 8.3+) | Сложный, жёсткая привязка к версии |
| **Custom XML** | Любая структура | Нужна обработка на стороне 1С |
| **HTTP API (OData)** | Прямое создание документов | Нужна публикация базы 1С |

**Рекомендация:** Custom XML + обработка загрузки в 1С.

Причины:
- CommerceML НЕ поддерживает проводки (Дт/Кт) — а вам нужно передавать готовые проводки
- EnterpriseData слишком привязан к конкретной конфигурации 1С
- HTTP API (OData) — хорош, но требует публикации базы 1С в сеть
- Custom XML — максимальная гибкость, обработку пишем один раз

### 4.4. Откат (Rollback)

**Правило:** 1С НЕ правится вручную. Если ошибка:

```
1. Получили error из 1С → status = 'error' в слое 3
2. Диагностика: AI или менеджер определяет причину
3. Исправление в слое 2 (корректировка документа)
4. Повторная верификация
5. Генерация нового чистового документа (слой 3)
6. Повторный экспорт

Если документ уже проведён в 1С и нужно отменить:
1. Отправляем команду "отменить проведение" через XML
2. 1С отменяет проведение, подтверждает в feedback
3. Экспортируем исправленный документ
4. 1С проводит новый документ
```

## 10. КРИТИЧЕСКОЕ УТОЧНЕНИЕ: ПРОВОДКИ В 1С

Исследование интеграции с 1С показало важный нюанс:

### Не передавать готовые проводки (Дт/Кт) в 1С

**Антипаттерн (как описано в промпте):**
```xml
<!-- НЕ НАДО: готовые проводки в XML -->
<Проводка>
  <Дебет>50.01</Дебет>
  <Кредит>90.01.1</Кредит>
  <Сумма>42300.00</Сумма>
</Проводка>
```

**Правильный подход:**
```xml
<!-- ПРАВИЛЬНО: передать документ, 1С сама сделает проводки -->
<DocumentObject.ОтчётОРозничныхПродажах>
  <Товары>
    <Row>
      <Номенклатура>Бензин АИ-92-К5</Номенклатура>
      <Количество>2459.80</Количество>
      <Цена>54.00</Цена>
      <Сумма>132829.20</Сумма>
      <СтавкаНДС>НДС20</СтавкаНДС>
    </Row>
  </Товары>
  <Оплата>
    <Row><ВидОплаты>Наличные</ВидОплаты><Сумма>85400.00</Сумма></Row>
    <Row><ВидОплаты>Платёжная карта</ВидОплаты><Сумма>142300.00</Сумма></Row>
  </Оплата>
</DocumentObject.ОтчётОРозничныхПродажах>
```

**Почему:**
- 1С имеет свою учётную политику и автоматически формирует проводки при проведении документов
- Счета учёта в БП 3.0 хранятся в регистре "СчетаУчётаНоменклатуры" — 1С подставляет их сама
- Передача проводок напрямую ломает целостность учёта в 1С
- Единственное исключение: документ "Операция" для ручных корректировок

**Как это влияет на архитектуру:**

Accounting Kernel (слой 3) по-прежнему формирует проводки, но использует их **для управленческого учёта и контроля**, а не для передачи в 1С:

```
Слой 2 (verified) → Accounting Kernel
                     ├── Проводки → для управленческих отчётов TradeFrame
                     ├── Проводки → для контроля (сравнить с 1С после проведения)
                     └── XML документ (без проводок) → 1С:БП
                                                       └── 1С сама формирует проводки
                                                       └── Feedback → сравниваем
```

### Рекомендуемый формат обмена с 1С

| Аспект | Решение |
|--------|---------|
| Основной транспорт | Файловый обмен (XML в папку) — надёжный, работает при выключенной 1С |
| Дополнительно | HTTP-сервис 1С — для справочников, статусов, оперативных запросов |
| Формат XML | Custom (не CommerceML) — CommerceML не покрывает специфику ГСМ |
| Проводки | НЕ передавать, 1С формирует сама при проведении |
| Обратная связь | Файл квитанции: статус, номер 1С, ошибки |
| Идемпотентность | Уникальный ИдСообщения в каждом XML |
| Справочники | Mapping по кодам, НЕ по наименованиям |

### Типичные ловушки интеграции с 1С

1. **Справочники:** Использовать коды (AI92K5), не наименования ("Бензин АИ-92") — наименования могут отличаться
2. **Округление НДС:** Считать построчно, итог = сумма строк (не наоборот)
3. **Кодировка:** Всегда UTF-8 с BOM для файлового обмена
4. **Нумерация:** Передавать как "внешний номер", 1С присваивает свой
5. **Закрытый период:** Проверять дату закрытия перед отправкой документа
6. **Пересчёт литры/кг:** Хранить ОБА значения + плотность + температуру

## 11. ОТВЕТ НА ГЛАВНЫЙ ВОПРОС: AI-Native учёт в бизнесе

### Почему трёхслойная архитектура — это правильный фундамент для будущего

Ваш вопрос: _"Правильный ли это подход с учётом появления AI-агентов к учёту в бизнесе?"_

**Да, и вот почему это не просто "хороший подход", а единственный правильный:**

#### 1. AI нужна структура, не интерфейс

Традиционный учёт:
```
Человек → UI (кнопки, формы) → Данные → Отчёты
```

AI-native учёт:
```
Данные → API → AI-агент → Данные → Человек (контроль) → Отчёты
```

Ваша архитектура убирает "UI-зависимость" и даёт AI прямой доступ к данным через API. Это фундаментально правильно.

#### 2. JSONB = самодокументируемость для AI

Реляционная схема с 50 таблицами и 200 FK — кошмар для AI (нужен огромный контекст).
JSONB metadata в одной таблице documents — AI читает структуру прямо из данных:

```json
{"shift_number": 6890, "payments": {"cash": 42300, "card": 78200}}
```

AI сразу понимает, что это за данные, без описания схемы.

#### 3. Декларативные правила = AI может их создавать

Если сверки — это код (Python-функции), AI может их только читать.
Если сверки — это конфигурация (YAML/JSON), AI может их **создавать и модифицировать**:

```
AI: "Я заметил, что на АЗС-208 расхождение по ДТ всегда в ночную смену.
     Предлагаю добавить правило: если расхождение <0.2% + ночная смена + ДТ → auto_ok"
```

#### 4. Document Graph = AI рассуждает о причинах

Без графа: "Расхождение 500₽ по кассе" — AI может только сообщить.
С графом: "Расхождение 500₽ → чек №142 был отменён → возврат не прошёл через банк → нужно повторить refund" — AI **находит причину и предлагает действие**.

#### 5. Иммутабельность = AI может учиться на истории

Append-only RAW данные + полный audit trail корректировок = идеальный тренировочный набор для AI:
- "Какие расхождения обычно оказываются ложными?"
- "Какие корректировки менеджер делает чаще всего?"
- "Есть ли сезонные паттерны потерь?"

### Что делает эту архитектуру по-настоящему AI-native

| Свойство | Классический учёт | Ваша архитектура | Почему важно для AI |
|----------|-------------------|------------------|---------------------|
| Данные | Спрятаны в UI | Доступны через API | AI работает с API |
| Схема | Жёсткая (50+ таблиц) | Гибкая (JSONB) | AI читает данные без описания |
| Правила | Захардкожены | Декларативные | AI может создавать/менять |
| Связи | Неявные (FK) | Явный граф | AI рассуждает о причинах |
| История | Перезаписывается | Иммутабельная | AI учится на истории |
| Действия | Только человек | AI с порогом уверенности | Автоматизация 80% рутины |

### Это подход к ЛЮБОМУ учёту, не только ГСМ

```
ГСМ:      STS API → RAW → Documents → Reconciliation → Clean → 1С
Зарплата: 1С ЗУП  → RAW → Documents → Reconciliation → Clean → 1С
Аренда:   Договор → RAW → Documents → Reconciliation → Clean → 1С
Продажи:  Касса   → RAW → Documents → Reconciliation → Clean → 1С
```

Одна архитектура, разные загрузчики + парсеры + правила сверки. AI-агент один на все модули — он понимает JSONB metadata любого типа документа.

### Куда это ведёт через 2-3 года

```
Сегодня:   AI предлагает → Человек подтверждает → Система исполняет
Через год:  AI исполняет типовое → Человек проверяет исключения
Через 2-3:  AI ведёт полный учёт → Человек задаёт вопросы и утверждает отчётность
```

Ваша архитектура готова ко всем трём этапам. Это не переделка — это постепенное увеличение автономности AI-агента через конфигурацию (пороги уверенности, разрешённые действия)
