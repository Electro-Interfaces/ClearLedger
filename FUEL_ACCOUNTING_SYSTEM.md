# Система учёта ГСМ

## Назначение

Модуль учёта ГСМ в рамках платформы TradeFrame — единая система контроля движения топлива на сети АЗС. Охватывает полный цикл: от приёмки топлива до продажи клиенту, с автоматическим выявлением расхождений.

Система организует четыре контура учёта, каждый из которых решает свою задачу, но работает на общих данных.

---

## Первый этап: Сквозная сверка всех разрезов учёта

### Идея

Прежде чем строить бухгалтерский и налоговый контуры, нужно навести порядок в данных. Задача первого этапа — **свести все источники данных в единую картину и убедиться, что цифры сходятся**.

Ключевой принцип: **мы готовим документы до выгрузки в 1C на чистовую**. В 1C не должно быть никаких ручных корректировок — туда уходят уже выверенные, согласованные данные. Бухгалтер в 1C только проводит готовые документы, не исправляя суммы и количества.

### Разрезы учёта (источники данных для сверки)

На каждой АЗС за смену происходят операции, которые фиксируются в разных системах. Все эти данные должны сойтись друг с другом:

| Разрез | Что фиксирует | Источник | Ключевые поля |
|--------|--------------|----------|---------------|
| **Операции на станции** | Каждая транзакция отпуска топлива | STS API `/v2/transactions` | Литры, сумма, вид топлива, способ оплаты, время, ТРК, пистолет |
| **Касса (ККТ)** | Фискальные чеки, наличные и безнал | STS API `/v2/info` (cash_sum, bank_sum) | Сумма наличных, сумма безнала за смену |
| **Банковские карты** | Эквайринговые операции | STS API (тип оплаты = карта) + банковский реестр | Сумма, количество операций, номер терминала |
| **Топливные карты** | Корпоративные операции | TradeCorp API | Литры, сумма, номер карты, контрагент |
| **Онлайн-заказы** | Мобильные платежи (СБП, приложение) | MSTO API | Литры, сумма, номер заказа, статус |
| **ТТН (приходы)** | Поступления топлива от поставщика | STS API `/v1/report/receipts` | Объём/масса по документу, объём/масса факт, плотность, поставщик |
| **Сменные отчёты** | Итоги смены: продажи, остатки, поступления | STS API `/v1/shift_report` | Реализация по видам, остатки резервуаров, показания ТРК, инкассация |

### Матрица сверки

Каждая пара разрезов сверяется по определённому показателю:

```
                  Операции   Касса   Банк.карты  Топл.карты  Онлайн   ТТН    Смен.отчёт
                  станции    (ККТ)                            заказы  приходы
Операции станции    —         ₽        ₽           л/₽        л/₽      —       л/₽
Касса (ККТ)         ₽         —        ₽безнал      —          —       —       ₽
Банк.карты          ₽        ₽безнал    —           —          —       —       ₽безнал
Топл.карты         л/₽        —         —           —          —       —       л/₽
Онлайн-заказы      л/₽        —         —           —          —       —       л/₽
ТТН приходы         —         —         —           —          —       —       л/кг
Сменный отчёт      л/₽        ₽       ₽безнал     л/₽        л/₽     л/кг      —
```

**Сменный отчёт — центральный узел.** Он содержит итоги по всем разрезам за смену и является основой для формирования документов в 1C.

### Что сверяем конкретно

**1. Операции станции ↔ Сменный отчёт**
- Сумма литров по виду топлива = реализация в сменном отчёте
- Сумма рублей по виду оплаты = выручка в сменном отчёте
- Количество операций

**2. Касса (ККТ) ↔ Сменный отчёт**
- cash_sum = сумма наличных в смене
- bank_sum = сумма безнала в смене
- Итого кассы = итого выручки смены

**3. Банковские карты ↔ Касса**
- Сумма по эквайрингу из банковского реестра = bank_sum из ККТ
- При расхождении — проверяем отмены, возвраты, техсбои

**4. Топливные карты ↔ Операции станции**
- Каждая операция по топливной карте (TradeCorp) должна найти пару в транзакциях станции (STS)
- Сверка по: литры, сумма, время (с допуском ±5 мин)

**5. Онлайн-заказы ↔ Операции станции**
- Каждый заказ MSTO должен найти пару в транзакциях станции
- Сверка по: литры, сумма, номер ТРК, время

**6. ТТН приходы ↔ Сменный отчёт**
- Объём по документу ↔ объём факт (отклонение в пределах нормы)
- Масса по документу ↔ масса факт
- Количество ТТН за смену

**7. Остатки резервуаров**
- Остаток на конец смены (датчик) ↔ расчётный остаток (начало + приход − расход)
- Расхождение = потери (в пределах или сверх нормы)

### Результат сверки за смену

По итогу сверки каждой смены формируется **протокол**:

```
Смена №6890 | АЗС 208 | 20.02.2026 07:31 — 20.02.2026 19:45

РЕАЛИЗАЦИЯ
  АИ-92:  1 240 л  ×  52.50 ₽  =  65 100.00 ₽   ✓ сходится
  АИ-95:    890 л  ×  56.80 ₽  =  50 552.00 ₽   ✓ сходится
  ДТ:       670 л  ×  58.20 ₽  =  38 994.00 ₽   ✓ сходится

СПОСОБЫ ОПЛАТЫ
  Наличные:     42 300.00 ₽  (ККТ: 42 300.00 ₽)   ✓
  Банк.карты:   78 200.00 ₽  (ККТ: 78 200.00 ₽, Банк: 78 200.00 ₽)   ✓
  Топл.карты:   24 146.00 ₽  (TradeCorp: 24 146.00 ₽)   ✓
  Онлайн:       10 000.00 ₽  (MSTO: 10 000.00 ₽)   ✓

ПОСТУПЛЕНИЯ
  ТТН №1234: АИ-92, док 12 000 л / факт 11 980 л  (−0.17%)   ✓ в норме

ОСТАТКИ РЕЗЕРВУАРОВ
  Рез.1 АИ-92: расчёт 14 200 л / факт 14 180 л  (−0.14%)   ✓ в норме
  Рез.2 ДТ:    расчёт 18 500 л / факт 18 460 л  (−0.22%)   ✓ в норме

СТАТУС: ✓ Смена выверена, готова к выгрузке в 1C
```

Если есть расхождения — смена получает статус «Требует проверки» с указанием конкретных проблем. Оператор или менеджер разбирает причину, вносит корректировку (например, отмена операции, повторный замер), и только после этого смена переходит в статус «Выверена».

## Подготовка документов для 1C

### Принцип: «чистовик до выгрузки»

```
TradeFrame                                    1C:Бухгалтерия

Данные ──→ Сверка ──→ Корректировка ──→ Документ ──→ Проведение
(сырые)    (авто)     (если нужна)      (чистовой)   (без правок)
```

В 1C уходят **только выверенные документы**. Бухгалтер не исправляет суммы, не ищет расхождения — он проводит готовые документы.

### Какие документы формируются

За каждую выверенную смену TradeFrame формирует пакет документов:

| Документ 1C | Основание | Что содержит |
|-------------|-----------|--------------|
| **Отчёт о розничных продажах** | Сменный отчёт | Реализация по видам топлива, суммы по способам оплаты |
| **Приходная накладная** | ТТН | Оприходование топлива от поставщика (объём, цена, сумма, НДС) |
| **Акт расхождений при приёмке** | ТТН vs Факт | Если факт ≠ документ: недостача или излишек при приёмке |
| **Операция инкассации** | Кассовые данные | Сумма сданных наличных |
| **Реестр корпоративных операций** | TradeCorp | Операции по топливным картам для выставления счетов контрагентам |
| **Реестр онлайн-операций** | MSTO | Операции через мобильные платежи |
| **Акт списания потерь** | Расчёт потерь | Потери в пределах/сверх нормы за период |

### Статусы документов

```
Черновик → На проверке → Выверен → Выгружен в 1C → Проведён в 1C
```

- **Черновик** — документ сформирован автоматически из данных смены
- **На проверке** — есть расхождения, требуется вмешательство
- **Выверен** — все сверки пройдены, документ готов к выгрузке
- **Выгружен в 1C** — отправлен в 1C (XML)
- **Проведён в 1C** — подтверждение из 1C, что документ проведён

## Четыре контура учёта

### 1. Технологический учёт

**Задача:** отвечать на вопрос «сколько топлива физически есть и куда оно делось».

Данные поступают автоматически с оборудования АЗС:
- **Резервуары** — уровень, объём, масса, температура, плотность, уровень воды (датчики, каждые 5-10 минут)
- **ТРК** — показания счётчиков при каждой операции отпуска
- **Приёмка** — фактические замеры при сливе из бензовоза (объём, масса, плотность, температура)

На этом уровне считаем:
- Фактические остатки в каждом резервуаре
- Объём отпуска по каждой ТРК за смену
- Факт поступления при каждой поставке
- **Потери** — разница между расчётным остатком (было + пришло − продано) и фактическим замером

Нормативы потерь регламентированы приказом Минэнерго №527 и ГОСТ Р 8.595-2004.

### 2. Управленческий учёт

**Задача:** давать информацию для принятия решений.

- **Маржа** по каждому виду топлива = цена продажи − закупочная цена
- **Оборачиваемость** ёмкости
- **Прогноз точки заказа**
- **Потери в рублях**
- **KPI по станциям** — объём продаж, средний чек, доля безналичных платежей

### 3. Бухгалтерский учёт

**Задача:** вести учёт по правилам ПБУ/ФСБУ для официальной отчётности.

Переводит операции с топливом в бухгалтерские проводки:
- **Оприходование** → Дт 41 Кт 60
- **Реализация** → Дт 50/51/62 Кт 90
- **Себестоимость** → Дт 90.2 Кт 41
- **Потери в норме** → Дт 44 Кт 41
- **Потери сверх нормы** → Дт 94 Кт 41
- **Инвентаризация** → сличительные ведомости

Все проводки формируются в TradeFrame на основании выверенных данных и выгружаются в 1C.

### 4. Налоговый учёт

**Задача:** корректно рассчитать налоговую базу.

- **НДС** — входящий (по ТТН) и исходящий (по продажам)
- **Нормативные потери** — уменьшают базу по налогу на прибыль (НК РФ ст. 254 п.7)
- **Сверхнормативные потери** — не уменьшают базу
- Данные для книги покупок и книги продаж

## Источники данных

| Источник | Что даёт | Протокол |
|----------|---------|----------|
| STS API (pos.autooplata.ru) | Резервуары, ТРК, смены, транзакции, поступления | REST API |
| ККТ (фискальный регистратор) | Фискальные чеки, суммы наличных/безнала | Через STS |
| TradeCorp | Корпоративные операции (топливные карты) | REST API |
| MSTO | Онлайн-заказы (мобильные платежи) | REST API |
| Supabase | Справочники (сети, станции, пользователи, пороги) | REST API |
| 1C:Бухгалтерия | Бухгалтерские данные (импорт/экспорт) | XML |

## Движение топлива (полный цикл)

```
Поставщик → ТТН → Приёмка на АЗС → Резервуар → ТРК → Клиент → ККТ
   │           │         │              │          │              │
   Закупка   Документ  Факт замера   Хранение   Отпуск       Оплата
   цена      объём/    объём/масса   остатки    литры        сумма
             масса     плотность     потери     стоимость    способ
```

## Роли и доступ

| Роль | Что видит | Что делает |
|------|-----------|-----------|
| Оператор АЗС | Свою станцию | Подтверждает приёмку, закрывает смену |
| Менеджер | Все станции региона | Разбирает расхождения, утверждает смены |
| Руководитель сети | Вся сеть | Полная аналитика, контроль потерь |
| Бухгалтер | Выверенные документы | Выгружает в 1C, проводит документы |

## Текущее состояние в TradeFrame

Что уже работает:
- Остатки резервуаров в реальном времени
- Транзакции продаж с разбивкой по видам топлива и способам оплаты
- Поступления по ТТН (документальные и фактические данные)
- Сменные отчёты с полной детализацией
- Книжные остатки (расчёт: было + пришло − продано)
- Сверка корпоративных операций (TradeCorp и MSTO)
- Экспорт в 1C XML (базовый)
- Калибровочные таблицы резервуаров

Что реализуем в рамках первого этапа:
- Сквозная сверка всех 7 разрезов учёта
- Протокол сверки за смену с автоматическим статусом
- Формирование чистовых документов для 1C
- Статусная модель документов (черновик → выверен → выгружен → проведён)
- Расчёт потерь (нормативных и фактических)
