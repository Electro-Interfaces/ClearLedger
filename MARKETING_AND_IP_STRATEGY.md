## Ключевой принцип

**AI-обработка — наша тайна.** Клиент получает результат, но не знает КАК он получен. Нейросеть настроена на наши знания и умения. Клиент пользуется только результатами нашего труда, не инструментами.

## 1. Архитектура защиты: «Умная коробка + облачный мозг»

### Две схемы инсталляции

```
Схема A: On-Premise (у клиента)              Схема B: Облако (у нас)
┌──────────────────────────────┐              ┌──────────────────────────────┐
│ Контейнер клиента            │              │ Наш сервер                   │
│ ├─ UI (React SPA)            │              │ ├─ UI (React SPA)            │
│ ├─ API (FastAPI)             │              │ ├─ API (FastAPI)             │
│ ├─ PostgreSQL                │              │ ├─ PostgreSQL                │
│ ├─ Layer 1 (RAW — файлы)    │              │ ├─ Layer 1 (RAW — файлы)    │
│ ├─ Layer 1a (staging)        │              │ ├─ Layer 1a (staging)        │
│ └─ Layer 2 (public — клиент) │              │ └─ Layer 2 (public — клиент) │
│                              │              │             ↕                │
│ Только механический парсинг  │              │ AI-обработка ЗДЕСЬ           │
│ (PyMuPDF, openpyxl, lxml)    │              │ (тот же сервер, локально)    │
└──────────────┬───────────────┘              └──────────────────────────────┘
               │ sync (mTLS, шифрованный канал)
               ▼
┌──────────────────────────────┐
│ Наш облачный AI-сервер       │
│ ├─ Skills Engine             │
│ ├─ LLM (Claude / GPT / etc) │
│ ├─ Справочники + нормативы   │
│ ├─ Лицензионный сервер       │
│ └─ Телеметрия + аудит        │
└──────────────────────────────┘
```

### Принцип разделения

| Компонент | Контейнер клиента | Облачный AI-сервер |
|-----------|:-:|:-:|
| UI (React SPA) | ✅ | — |
| API (FastAPI) | ✅ | — |
| PostgreSQL (данные) | ✅ | — |
| Механический парсинг (PyMuPDF, openpyxl) | ✅ | — |
| Pipeline L1→L1a (detect, extract, dedup) | ✅ | — |
| AI-классификация | — | ✅ |
| AI-нормализация и сверка | — | ✅ |
| Skills (правила учёта, навыки) | — | ✅ |
| Справочники (НСИ, нормативы) | кэш subset | ✅ (мастер) |
| Лицензионный сервер | — | ✅ |

**Контейнер клиента — «глупый».** Умеет только:
1. Принять файл (email, API, drag&drop)
2. Механически разобрать (OCR, текст, таблицы)
3. Отправить extracted_text в облако
4. Получить результат (классификация, нормализация, Layer 2)
5. Показать результат клиенту

Весь «ум» — в облаке. Даже при on-premise контейнер без облака = пустая оболочка с механическим парсером. Нет AI → нет классификации → нет сверки → нет ценности.

### Что уходит в облако (и что НЕ уходит)

| Уходит в облако | НЕ уходит в облако |
|---|---|
| `masked_text` (текст после PII-маскирования) | Оригинальные файлы (PDF, сканы) |
| `extracted_fields` (метаданные, маскированные) | ПДн в открытом виде (ФИО, паспорта, телефоны) |
| `company_profile` (тип бизнеса) | Логины/пароли |
| `instance_id` (идентификатор) | Данные Layer 2 (результаты) |

Облако обрабатывает маскированный текст → возвращает решение → контейнер подставляет оригиналы обратно.

### Privacy-hard: маскирование ПДн до отправки (MVP)

**Проблема:** `extracted_text` часто содержит ПДн и коммерческую тайну клиента — ФИО водителей, номера карт, телефоны, адреса, реквизиты контрагентов. Даже без исходных файлов это «данные ушли в облако». Для корпоративных клиентов и регуляторики (152-ФЗ) это deal-breaker.

**Решение: PII-маскер в контейнере (до отправки в облако).**

```
Контейнер клиента                         Облако
┌───────────────────────┐                 ┌──────────────────┐
│ extracted_text:       │                 │                  │
│ "ТТН №245, водитель   │   PII-маскер   │  masked_text:    │
│  Иванов И.И.,         │ ────────────→   │  "ТТН №245,      │
│  ИНН 7707083893,      │                 │   водитель       │
│  карта *4532"         │                 │   [PERSON_1],    │
│                       │                 │   ИНН [INN_1],   │
│ pii_map (локально):   │                 │   карта [CARD_1]"│
│ { PERSON_1: "Иванов   │                 │                  │
│   И.И.", ... }        │   ← результат   │  AI обрабатывает │
│                       │   + обратная    │  маскированный   │
│ Восстановленный       │   подстановка   │  текст           │
│ результат с ПДн       │                 │                  │
└───────────────────────┘                 └──────────────────┘
```

**Что маскируется (regex + NER для российских паттернов):**

| Тип ПДн | Паттерн | Замена |
|---------|---------|--------|
| ФИО | `Иванов Иван Иванович` | `[PERSON_1]` |
| ИНН | `\d{10,12}` в контексте | `[INN_1]` |
| СНИЛС | `\d{3}-\d{3}-\d{3} \d{2}` | `[SNILS_1]` |
| Телефон | `+7...`, `8(9...` | `[PHONE_1]` |
| Банковская карта | `\d{4} \d{4}...`, `*\d{4}` | `[CARD_1]` |
| Р/с, к/с | `\d{20}` в контексте | `[ACCOUNT_1]` |
| Паспорт | `\d{4} \d{6}` | `[PASSPORT_1]` |
| Email | стандартный regex | `[EMAIL_1]` |

**Ключевое:** `pii_map` (маппинг `[PERSON_1]` → `Иванов И.И.`) хранится ТОЛЬКО в контейнере клиента. Облако никогда не видит реальные ПДн. AI работает с маскированным текстом — для классификации и нормализации достаточно структуры документа, не конкретных ФИО.

**Три режима privacy (настраивается клиентом):**

| Режим | Что маскируется | Для кого |
|-------|----------------|----------|
| `standard` | Только критичные ПДн (ФИО, паспорта, карты) | Большинство клиентов |
| `strict` | Все ПДн + реквизиты контрагентов + суммы | Банки, госсектор, крупные корпорации |
| `local-first` | В облако уходят только исключения (расхождения, нераспознанное). 90%+ обрабатывается локально rules-first | Enterprise с жёсткой политикой безопасности |

Режим `local-first` опирается на уже заложенный принцип «Код vs AI» из [PLAN.md](./PLAN.md): детерминированные операции (парсинг, математика, валидация) выполняются локально, AI подключается только при расхождениях — с минимальным контекстом.

## 2. Механизмы защиты IP (7 уровней)

### Сводная таблица

| # | Уровень | Механизм | Что защищает | Стоимость обхода |
|---|---------|----------|--------------|------------------|
| 1 | **Архитектурный** | AI ТОЛЬКО в облаке. В контейнере нет моделей, Skills, весов, промптов | Невозможно извлечь AI — его физически нет в контейнере | Написать свой AI с нуля |
| 2 | **Лицензионный** | `INSTANCE_ID` + `CLOUD_API_KEY` привязаны к ИНН + MAC-адрес + hostname | Невозможно клонировать контейнер и запустить второй | Подделать ИНН + MAC (уголовное) |
| 3 | **Криптографический** | mTLS (mutual TLS) контейнер↔облако. Клиентский сертификат вшит в образ | Невозможно подставить свой сервер вместо нашего | Украсть приватный ключ из образа |
| 4 | **PII-маскирование** | В облако уходит `masked_text` после маскирования ПДн. `pii_map` остаётся ТОЛЬКО в контейнере | ПДн клиента не покидают его контур. 152-ФЗ compliant | Реверсить протокол маскирования |
| 5 | **Юридический** | Лицензионный договор: запрет реверс-инжиниринга, штраф 5× годовой стоимости | Юридический барьер + компенсация ущерба | Нарушение договора (суд) |
| 6 | **Heartbeat** | Проверка лицензии каждые 24ч. Без связи >7 дней → read-only mode | Нельзя отключить облако и продолжить полноценно работать | Подменить ответ heartbeat (mTLS мешает) |
| 7 | **Skills как сервис** | Навыки (правила, нормативы, учётные политики) хранятся ТОЛЬКО в облаке | Без навыков контейнер бесполезен: не знает правил учёта | Написать свои навыки (годы экспертизы) |

### Детализация ключевых механизмов

#### 2.1 Архитектурная изоляция (Уровень 1)

Контейнер `clearledger:latest` содержит:
- Python-код (FastAPI, pipeline)
- React SPA (frontend)
- PostgreSQL-миграции
- Механические парсеры (PyMuPDF, openpyxl, python-docx)

Контейнер **НЕ содержит:**
- Промпты для LLM
- Skills (навыки учёта)
- Модели / веса нейросетей
- Логику принятия решений (classify, normalize, reconcile)
- Справочники нормативов (ГОСТ, Минэнерго, ПБУ/ФСБУ)

Даже если клиент получит полный доступ к Docker-образу, он найдёт только механический парсер и HTTP-клиент для облака.

#### 2.2 Лицензионная привязка (Уровень 2)

```
При первом запуске:
1. Контейнер генерирует INSTANCE_ID = sha256(ИНН + MAC + hostname + salt)
2. Администратор регистрирует INSTANCE_ID на лицензионном сервере
3. Сервер выдаёт CLOUD_API_KEY (срок = срок подписки)
4. CLOUD_API_KEY хранится в контейнере, шифруется INSTANCE_ID

При каждом запросе к облаку:
- Header: X-Instance-ID: {INSTANCE_ID}
- Header: Authorization: Bearer {CLOUD_API_KEY}
- Облако проверяет: ключ валиден? Привязан к этому INSTANCE_ID? Подписка активна?
```

Если контейнер перенести на другой сервер (другой MAC/hostname) → INSTANCE_ID изменится → CLOUD_API_KEY не пройдёт проверку → нужна перерегистрация.

#### 2.3 Heartbeat и деградация (Уровень 6)

```
Heartbeat: POST https://license.clearledger.ru/api/heartbeat
Каждые 24 часа

Тело:
{
  "instance_id": "...",
  "version": "1.2.3",
  "documents_processed_24h": 47,
  "last_sync": "2026-02-27T10:00:00Z"
}

Ответ:
{
  "status": "active",           // active | grace | suspended
  "next_check": "2026-02-28T10:00:00Z",
  "features": ["case1", "case2", "case3"],
  "message": null
}
```

**Режимы деградации:**

| Состояние | Когда | Что работает | Что НЕ работает |
|-----------|-------|-------------|-----------------|
| `active` | Подписка оплачена, heartbeat <24ч | Всё | — |
| `grace` | Нет связи 1-7 дней | Просмотр, экспорт, парсинг в очередь | AI-обработка (копится в sync_queue) |
| `suspended` | Нет связи >7 дней ИЛИ подписка истекла | Только чтение (read-only) | Загрузка, обработка, экспорт |
| `terminated` | Договор расторгнут | Экспорт данных (30 дней) | Всё остальное |

Логика деградации встроена в контейнер. При `grace` — парсинг работает, документы копятся в `sync_queue`, как только связь восстанавливается — всё обрабатывается. При `suspended` — данные не теряются, но работа остановлена.

### Экономика подписки

**Почему подписка, а не продажа лицензии:**
1. **AI = постоянные расходы.** Каждый вызов Claude/GPT стоит денег. Разовая продажа не покрывает эксплуатацию.
2. **Skills обновляются.** Нормативы меняются, навыки дорабатываются — клиент получает обновления автоматически.
3. **Отключение = деградация.** Клиент не может «купить и забыть» — без облака контейнер бесполезен.
4. **Предсказуемый revenue.** MRR (Monthly Recurring Revenue) позволяет планировать развитие.

**Unit-экономика (пример тариф Business):**
```
Доход:          12 000 ₽/мес
Расход AI:       1 500 ₽/мес (~150 вызовов × ~10₽ средний)
Инфраструктура:    500 ₽/мес (доля от облачного сервера)
Поддержка:         500 ₽/мес (доля от ФОТ поддержки)
──────────────────────────────
Маржа:           9 500 ₽/мес (79%)
```

Стоимость AI-вызова оптимизируется:
- Prompt caching (TTL 300с для Skills — экономия 90% на повторных вызовах)
- Haiku для рутинных операций (~$0.0025/запрос)
- Sonnet/Opus только для сложных случаев
- Multi-model fallback: Claude → GPT → Gemini (по цене/качеству)

## 4. Маркетинговая стратегия

### Позиционирование

**Мы продаём результат (чистые документы), а не софт.**

Клиент не покупает «систему обработки документов». Клиент покупает:
- **Кейс 1:** «Все ваши документы проверены и подтверждены ДО попадания в 1С»
- **Кейс 2:** «Расхождения с контрагентами найдены СЕЙЧАС, а не при закрытии квартала»
- **Кейс 3:** «Закрытие периода пройдёт без сюрпризов — вы видите картину заранее»

Аналогия для клиента: **«ClearLedger — это ваш идеальный бухгалтер-стажёр, который проверяет каждый документ, никогда не устаёт и работает 24/7. Только он проверяет 1000 документов за 10 минут.»**

### Что НЕ показываем клиенту

| Скрыто | Почему |
|--------|--------|
| Архитектура AI (промпты, Skills, цепочки) | Ядро IP. Знание = возможность воспроизвести |
| Содержание Skills (правила учёта) | Годы наработок. Конкурент скопирует за месяц |
| Механика принятия решений | Клиент должен доверять результату, а не контролировать процесс |
| Стоимость API-вызовов | Клиент не должен считать нашу маржу |
| Мультимодельный fallback | Технический бэкенд — не забота клиента |

### Каналы продаж

#### 4.1 Отраслевые вертикали (основной)

Стратегия: **глубокое проникновение в одну вертикаль → расширение.**

```
ГСМ (розница) → ГСМ (опт) → Продуктовый ритейл → Стройматериалы → ...
```

Почему ГСМ первый:
- Высокая частота документов (ТТН, акты, счета-фактуры ежедневно)
- Естественные расхождения (усушка, утруска, температурные коэффициенты)
- Болезненное закрытие периода (акцизы, ретро-бонусы)
- Собственники мотивированы контролем (потери = прямые убытки)

#### 4.2 Партнёрская программа с 1С-франчайзи (20% комиссии)

1С-франчайзи — естественный канал:
- Они уже общаются с клиентами о бухгалтерских проблемах
- ClearLedger не конкурирует с 1С — дополняет
- 20% от подписки партнёру = 2 400 ₽/мес с Business → мотивация рекомендовать
- Партнёр делает интеграцию с конкретной 1С клиента → доп. доход

**Модель:**
```
Клиент платит нам 12 000 ₽/мес (Business)
Мы платим 1С-франчайзи 2 400 ₽/мес (20%)
Мы получаем 9 600 ₽/мес
Франчайзи получает пассивный доход + работу по интеграции (25 000 ₽ разово)
```

#### 4.3 Бухгалтерский аутсорс

Аутсорсинговые бухгалтерские компании — мультипликатор:
- Одна компания = 10-50 клиентов
- ClearLedger экономит им часы ручной работы
- Специальный тариф «Аутсорс» = 8 000 ₽/мес за клиента (скидка за объём)
- Они перепродают клиенту как часть своей услуги

#### 4.4 Контент-маркетинг

Контент для доверия и organic-трафика:
- **Кейс-стади:** «Как сеть из 5 АЗС сократила время закрытия периода с 5 дней до 4 часов»
- **Калькулятор потерь:** интерактивный инструмент «Сколько вы теряете на ручной проверке»
- **Вебинары:** «Типичные ошибки в учёте ГСМ» (полезный контент → лиды)
- **Telegram-канал:** еженедельные советы по учёту ГСМ

#### 4.5 Демо на реальных данных

Самый мощный аргумент — **показать результат на данных клиента:**
1. Клиент загружает 10-20 документов за последний месяц
2. ClearLedger обрабатывает за 5 минут
3. Находит 3-5 расхождений, которые клиент не замечал
4. «Вот это вы пропустили. За год это X рублей. Подписка стоит Y рублей.»

**Конверсия демо → подписка: целевой показатель 40%+** (потому что клиент видит реальные свои ошибки).

## 5. Конкурентные барьеры (moat)

### 5 слоёв защиты от конкурентов

| # | Барьер | Время на воспроизведение | Описание |
|---|--------|:---:|---|
| 1 | **Библиотека Skills** | 2+ года | Каждый клиент = новые правила, исключения, отраслевые нюансы. 100+ навыков = невоспроизводимо |
| 2 | **Данные для обучения** | 1+ год | Обезличенные паттерны ошибок, расхождений, типичных проблем. Чем больше клиентов → точнее AI |
| 3 | **Интеграции с 1С** | 6+ мес | XML-маппинги под каждую версию и конфигурацию. Нетиповые 1С = отдельная работа |
| 4 | **Отраслевая экспертиза** | 1+ год | Нормативы Минэнерго, ГОСТ Р 8.595-2004, отраслевые правила учёта ГСМ, специфика акцизов |
| 5 | **Switching cost** | — | Клиент привыкает к чистым данным. Возврат к ручной работе — психологически невозможен |

### Почему конкуренту трудно повторить

**1С добавляет AI?**
- У 1С фокус на учёт, не на предварительную обработку
- ClearLedger работает ДО 1С, не вместо
- Наша фора: 2-3 года наработок Skills + паттернов
- 1С вряд ли будет инвестировать в нишевой reconciliation (для них это периферия)

**Стартап копирует идею?**
- Skills = 2+ года наработки экспертизы (не код, а знания)
- Каждый клиент обогащает библиотеку (network effect)
- Интеграции с нетиповыми 1С — ручная работа, не масштабируется быстро
- Без отраслевой экспертизы AI будет давать неточные результаты

**Enterprise-платформа (BlackLine/Duco) спускается в рынок?**
- Они работают с SAP/Oracle, не с 1С
- Русский рынок для них маргинален
- Цена $50K-350K/год vs наши 12K-70K ₽/мес (разница в 50×)

### Эффект масштаба (network effect для Skills)

```
Клиент 1 (ГСМ розница)     → Skills: ТТН, акцизы, усушка
Клиент 2 (ГСМ опт)         → Skills: + ретро-бонусы, оптовые скидки
Клиент 3 (продукты)        → Skills: + сроки годности, маркировка
Клиент N                   → Skills: + ...
                                     ↓
                            Библиотека из 200+ навыков
                            Точность AI: 95%→97%→99%
                            Каждый новый клиент = быстрее онбординг
```

Каждый клиент делает платформу умнее для ВСЕХ (подробнее: [PLAN.md, секция 4 «Онбординг»](./PLAN.md#4-онбординг-вход-в-работающий-бизнес)).

## 6. Юридическая защита

### 6.1 Патент на способ (Роспатент)

**Что патентуем:** «Способ автоматизированной верификации и нормализации первичных документов с использованием AI-навыков и эталонных данных из учётных систем»

**Ключевые элементы формулы:**
1. Приём документов из множественных каналов
2. Механический парсинг → extracted_text
3. Передача extracted_text в облачный AI-сервер через mTLS
4. AI-обработка на основе Skills (навыков на естественном языке)
5. Сверка с эталонными данными из 1С клиента
6. Возврат нормализованных данных в контейнер

**Сроки:** подача заявки — Q2 2026, экспертиза — 12-18 мес.
**Стоимость:** ~80 000 ₽ (пошлины + патентный поверенный)

### 6.2 Товарный знак (ФИПС)

- **ClearLedger** — словесный товарный знак (класс МКТУ 42: SaaS, обработка данных)
- **Логотип** — комбинированный товарный знак
- **Сроки:** подача — Q1 2026, регистрация — 8-12 мес.
- **Стоимость:** ~50 000 ₽

### 6.3 Лицензионный договор (ст. 1235 ГК РФ)

ClearLedger поставляется как **SaaS-услуга**, а не как продажа ПО:

| Аспект | Формулировка |
|--------|-------------|
| Предмет | Предоставление неисключительного права использования ПО ClearLedger в режиме SaaS |
| Способ использования | Доступ через веб-интерфейс / контейнер с подключением к облачному сервису |
| Ограничения | Запрет: декомпиляция, реверс-инжиниринг, копирование, модификация, субцессия |
| Ответственность | Штраф за нарушение: 5× годовой стоимости подписки |
| Территория | Российская Федерация |
| Срок | Срок подписки (пролонгация) |

**Преимущества SaaS vs продажа лицензии:**
- Нет передачи экземпляра ПО → нет исчерпания прав (ст. 1272 ГК РФ)
- Контроль над обновлениями и отключением
- Не попадает под обязательную регистрацию в РРПО (реестр российского ПО) — при желании

### 6.4 Ноу-хау + коммерческая тайна (ФЗ-98)

**Что является ноу-хау:**
- Содержание Skills (навыки учёта, правила, промпты)
- Архитектура цепочки обработки (pipeline решений)
- Обученные модели и параметры fine-tuning
- Библиотека паттернов ошибок и расхождений

**Режим коммерческой тайны (ФЗ-98 от 29.07.2004):**
1. Определён перечень информации, составляющей КТ
2. Нанесена маркировка «Коммерческая тайна» на носители
3. Ограничен доступ (список допущенных лиц)
4. Учёт лиц, получивших доступ
5. Включены условия о КТ в трудовые договоры и NDA

### 6.5 Защита от сотрудников

| Документ | Для кого | Ключевые пункты |
|----------|---------|-----------------|
| **NDA** | Все сотрудники | Неразглашение КТ, штраф 1 000 000 ₽ за нарушение |
| **Неконкуренция** | Ключевые разработчики | 2 года после увольнения: запрет работы на конкурентов в сфере AI-reconciliation |
| **IP Assignment** | Все разработчики | Все результаты интеллектуальной деятельности принадлежат компании (ст. 1295 ГК РФ — служебные произведения) |
| **Трудовой договор** | Все | Пункт о КТ, режиме секретности, ответственности |

> **Примечание:** условие о неконкуренции в РФ не имеет прямой судебной поддержки (в отличие от США/UK), но работает как сдерживающий фактор + основание для переговоров.

## 7. Риски и митигация

### Матрица рисков

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|:-:|:-:|---|
| 1 | **Клиент отключается от облака** | Средняя | Высокое | Heartbeat 24ч → grace 7 дней → read-only. Контейнер без облака = парсер без AI |
| 2 | **Реверс-инжиниринг API** | Низкая | Критическое | mTLS + шифрование payload + договорной штраф 5× + нет исходников AI |
| 3 | **1С добавляет AI-reconciliation** | Средняя | Высокое | Фора 2-3 года + библиотека 100+ Skills + отраслевая глубина. 1С фокус ≠ pre-accounting |
| 4 | **Конкурент копирует концепцию** | Средняя | Среднее | Skills = 2+ года наработки. Патент на способ. Network effect (каждый клиент обогащает) |
| 5 | **Claude API дорожает** | Средняя | Среднее | Multi-model fallback: Claude → GPT → Gemini. Prompt caching снижает стоимость на 90% |
| 6 | **Утечка Skills через сотрудника** | Низкая | Критическое | NDA + КТ + IP Assignment + минимальные права доступа. Skills хранятся фрагментировано |
| 7 | **Регуляторные изменения (ПДн, 152-ФЗ)** | Низкая | Среднее | On-premise опция. В облако уходит текст, не ПДн. Обработка — в контуре клиента |
| 8 | **Клиент требует self-hosted AI** | Средняя | Среднее | Только Enterprise (70K+). Отдельный контракт. AI на сервере клиента, но Skills обновляются из облака |

### Детализация ключевых рисков

#### 7.1 Клиент отключается от облака

**Сценарий:** Клиент оплатил on-premise, получил контейнер, блокирует исходящие соединения к облаку.

**Защита (эшелонированная):**
1. Heartbeat каждые 24ч — без ответа через 7 дней → `suspended` (read-only)
2. Контейнер без AI бесполезен — может только парсить файлы, но не классифицировать, не сверять, не нормализовать
3. Skills не кэшируются локально — при каждом вызове загружаются из облака
4. `sync_queue` копит необработанные документы — растёт давление на оператора

**Результат:** Даже если клиент обойдёт heartbeat, он получит механический парсер без AI. Это не имеет коммерческой ценности.

#### 7.2 Claude API дорожает или становится недоступен

**Сценарий:** Anthropic повышает цены в 3× или вводит ограничения для российских клиентов.

**Защита:**
1. **Multi-model architecture:** код написан с абстракцией LLM-провайдера
2. **Fallback цепочка:** Claude (primary) → GPT-4o (secondary) → Gemini (tertiary)
3. **Prompt caching:** TTL 300с для Skills — 90% запросов идут из кэша (стоимость → ~10% от полной)
4. **Haiku для рутины:** 80% операций выполняет Haiku ($0.0025/запрос), Sonnet/Opus — только сложные случаи
5. **Планируемый local fallback:** open-source модель (Llama/Mistral) для базовых операций при недоступности облачных API

#### 7.3 Клиент требует полный self-hosted

**Сценарий:** Крупный клиент (Enterprise) требует, чтобы всё, включая AI, работало на его серверах.

**Решение:**
- Только тариф Enterprise (от 70 000 ₽/мес)
- Отдельный контракт с повышенной стоимостью
- AI-контейнер на сервере клиента, но:
  - Skills обновляются из нашего облака (pull, не push)
  - Телеметрия (анонимизированная) отправляется нам
  - Лицензионный heartbeat сохраняется
- Skills поставляются как зашифрованный пакет (не plaintext)
